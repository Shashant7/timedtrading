# X Queue — 2026-05 (30-day rolling)

> **Purpose:** machine-readable queue for the dumb cron runner. The runner pulls the next item where `status: queued` and `scheduled_for_et` is in the past, posts via the X API, then sets `status: posted`.
>
> **Source of authority:** *only* the marketing agent edits this file (human-authored copy). The runner is read-mostly — it can flip `status` and write the resulting tweet URL into `posted_url`, but **never** rewrites `copy`.
>
> **Refresh cadence:** weekly. Each Monday the agent revisits the next 7 days, swaps in fresh `live-trade` posts using the operator's screenshots + moments, and extends the tail of the queue by 7 days.
>
> **Format:** YAML-style numbered list (parseable). One item per post. All times in **ET**. Stable IDs (`q-YYYY-MM-DD-NN`) — never reused.
>
> **Categories** (used for analytics + rewrite tuning):
>
> - `daily-brief` — pre-market levels post (anchor template B1–B5)
> - `live-trade` — post-event trade callout (anchor template A1–A5)
> - `philosophy` — one-liner engine truth (anchor template C1–C5)
> - `social-proof` — quote card (anchor template D1–D5) **— gated on operator providing consented quotes**
>
> **Disclaimer flag** = `true` whenever the post body or image surfaces a number (price, %, $, win count). `false` is allowed only for pure-philosophy posts with no numerics.

---

## Operator setup (one-time, before the runner can post)

1. Operator confirms today's account / PnL numbers from `/proof.html` and edits anything labeled `{{token}}` in this file.
2. Operator collects consented testimonials → fills the `social-proof` slots (Q-D1 through Q-D5). Until then, `social-proof` rows below are marked `status: blocked-pending-quotes`.
3. Operator confirms image assets exist for every `visual_brief` filename (see column). Files live under `tasks/marketing/assets/x/` (operator handles upload to the X API runner storage).
4. Operator sets the runner's clock to ET. The runner only fires during the ET window 08:00 – 17:00 weekdays + 09:00 – 12:00 weekends.

## Runner contract (operator implements once)

The runner is intentionally dumb. Pseudocode:

```js
// every 30 min, ET-aware:
for (item of queue.filter(i => i.status === 'queued' && i.scheduled_for_et <= nowET())) {
  // any token of form {{...}} remaining ⇒ skip + alert operator
  if (item.copy.match(/\{\{[^}]+\}\}/)) { alertOperator(item.id, 'unfilled tokens'); continue; }
  const tweet = await postToX({ text: item.copy, media: item.visual_brief });
  item.status = 'posted';
  item.posted_url = tweet.url;
  item.posted_at = nowET();
  await saveQueue();
}
```

- **The runner never edits `copy`.** If a `{{token}}` is still present, the runner alerts the operator and skips the slot. The agent fills tokens during the weekly refresh; the operator fills `live-trade` tokens via a quick edit before the day's 4:30 PM ET slot.
- **`blocked-pending-quotes` rows are never auto-posted.** The runner skips them silently until the operator either supplies a consented quote (and flips `status` to `queued`) or deletes the row.
- **Visual brief filenames** under `tasks/marketing/assets/x/` are the canonical paths. The runner uploads them to X media before composing the tweet. If the file is missing, alert the operator and skip.
- **Rate limit:** the runner enforces no more than **3 posts per UTC day** across this queue (per canonical plan §4 cadence). Any 4th eligible item is held over to the next day.
